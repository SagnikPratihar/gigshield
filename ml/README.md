## рЯУУ Running Notebooks

Notebooks are meant to be run **in order** вАФ each one builds on the previous.

### Start Jupyter

```bash
# Make sure your venv is active
jupyter notebook
```

This opens Jupyter in your browser at `http://localhost:8888`.

---

### Notebook 1 вАФ `01_data_exploration.ipynb`

**What it does:**
- Loads raw weather and disruption data from `data/raw/`
- Plots rainfall frequency by city and month (shows monsoon seasonality)
- Shows disruption event distribution across zones
- Correlates disruption hours with estimated income loss
- Generates zone risk heatmaps

**Run all cells.** No model is trained here вАФ this is purely understanding the data.

**Key outputs:**
- City-wise disruption frequency chart
- Monthly rainfall distribution (proof of seasonality)
- Zone risk tier validation

---

### Notebook 2 вАФ `02_severity_classifier.ipynb`

**What it does:**
- Loads `data/processed/features_severity.csv`
- Trains a **Random Forest Classifier** to label each disruption event as:
  - `MILD` вАФ payout modifier 0.5√Ч
  - `MODERATE` вАФ payout modifier 1.0√Ч
  - `SEVERE` вАФ payout modifier 1.5√Ч
- Plots feature importance, confusion matrix, classification report

**Key inputs:**
```
disruption_type, severity_value, duration_hours,
zone_risk_tier, time_of_day, historical_avg_severity
```

**Key outputs:**
- Trained model saved to `models/severity_clf.pkl`
- F1-score target: > 0.85
- Feature importance plot

---

### Notebook 3 вАФ `03_loss_estimator.ipynb`

**What it does:**
- Loads disruption + worker income data
- Trains a **Gradient Boosting Regressor** to predict вВє income loss given a disruption
- Validates predictions against synthetic ground truth
- Plots actual vs predicted income loss

**Key inputs:**
```
disruption_type, severity_class, duration_hours,
worker_avg_hourly_income, zone_risk_tier, time_of_day
```

**Key outputs:**
- Trained model saved to `models/loss_estimator.pkl`
- R¬≤ target: > 0.80
- Residual plot (prediction error distribution)

---

### Notebook 4 вАФ `04_premium_model.ipynb`

**What it does:**
- Loads `data/processed/features_premium.csv`
- Trains an **XGBoost Regressor** to calculate weekly premium adjustment (¬± вВє)
- Shows which features drive premium up or down most
- Validates with worked examples (Bangalore non-monsoon vs Mumbai monsoon)

**Key inputs:**
```
zone_id, city_risk_tier, season_month,
forecast_rain_7day_mm, forecast_max_temp,
zone_flood_events_12m, zone_rain_events_12m,
worker_avg_daily_hours, worker_claim_score,
zone_aqi_7day_avg, zone_civic_risk_index
```

**Key outputs:**
- Trained model saved to `models/premium_model.pkl`
- MAE target: < вВє5 error
- SHAP feature importance plot

---

### Notebook 5 вАФ `05_fraud_detection.ipynb`

**What it does:**
- Loads `data/synthetic/synthetic_fraud_cases.csv`
- Trains an **Isolation Forest** anomaly detector
- Tunes the `contamination` parameter (expected % of fraud in real traffic)
- Plots fraud score distribution (legit vs fraud clusters)
- Validates against known fraud cases

**Key inputs:**
```
distance_worker_to_zone_km, worker_platform_status,
claim_frequency_30d, avg_claim_amount_30d,
hours_since_last_claim, device_id_match,
gps_trajectory_score, event_cluster_count
```

**Key outputs:**
- Trained model saved to `models/fraud_detector.pkl`
- Precision target: > 0.90 on fraud cases
- Score distribution plot (separates legit vs fraudulent)

## рЯФМ API Reference

All endpoints accept and return JSON. Spring Boot calls these internally вАФ workers never interact with this API directly.

---

### `GET /ml/health`
Check API status and which models are loaded.

**Response:**
```json
{
  "status": "UP",
  "models_loaded": 4
}
```

---

### `POST /ml/severity`
Classify how severe a disruption event is.

**Request:**
```json
{
  "disruption_type": "HEAVY_RAIN",
  "severity_value": 42.3,
  "duration_hours": 2.5,
  "zone_risk_tier": 4,
  "time_of_day": 19,
  "historical_avg_severity": 28.0
}
```

**Response:**
```json
{
  "severity_class": "MODERATE",
  "payout_modifier": 1.0,
  "confidence": 0.87
}
```

---

### `GET /ml/premium`
Get the AI-adjusted weekly premium for a worker.

**Request params:**
```
zone_id, city_risk_tier, season_month, forecast_rain_7day,
forecast_max_temp, zone_flood_events_12m, zone_rain_events_12m,
worker_avg_daily_hours, worker_claim_score, zone_aqi_avg, civic_risk_index
```

**Response:**
```json
{
  "base_premium": 59.00,
  "adjustment": -14.50,
  "final_premium": 44.50,
  "risk_level": "LOW",
  "top_factors": ["low_zone_flood_history", "clear_forecast", "clean_claim_record"]
}
```

---

### `POST /ml/fraud`
Score a claim for fraud risk before processing payout.

**Request:**
```json
{
  "worker_id": 57,
  "distance_to_zone_km": 1.2,
  "platform_status": "ACTIVE",
  "claim_frequency_30d": 2,
  "avg_claim_amount_30d": 310.0,
  "hours_since_last_claim": 168,
  "device_id_match": true,
  "gps_trajectory_score": 0.1,
  "event_cluster_count": 45
}
```

**Response:**
```json
{
  "fraud_score": 14.2,
  "decision": "AUTO_APPROVE",
  "flags": []
}
```

Possible `decision` values:
- `AUTO_APPROVE` вАФ score 0вАУ30, payout proceeds immediately
- `AUTO_APPROVE_FLAGGED` вАФ score 31вАУ69, payout proceeds + logged for review
- `HOLD_FOR_REVIEW` вАФ score 70вАУ100, payout held pending admin approval

---

### `POST /ml/forecast`
Estimate income loss for a disruption event (used for payout calculation).

**Request:**
```json
{
  "disruption_type": "HEAVY_RAIN",
  "severity_class": "MODERATE",
  "duration_hours": 2.5,
  "worker_avg_hourly_income": 186.67,
  "zone_risk_tier": 4,
  "time_of_day": 19
}
```

**Response:**
```json
{
  "estimated_loss_inr": 280.0,
  "confidence_interval": [245.0, 315.0]
}
```

## рЯІ™ Synthetic Data

We don't have real claims history yet вАФ GigShield is a new product. So we generate **realistic synthetic training data** using:

- Real historical weather data from IMD and OpenWeatherMap (public datasets)
- Known income loss patterns documented in gig economy research (20вАУ30% loss during disruptions)
- Realistic fraud scenario patterns based on documented insurance fraud cases

This is **standard industry practice** вАФ insurance actuaries always use synthetic or proxy data when launching new parametric products. There is no real claims history to learn from at launch.

All assumptions used to generate synthetic data are documented in:
```
data/synthetic/GENERATION_NOTES.md
```
